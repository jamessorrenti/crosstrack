# crosstrack

A Google Sheets tracking ecosystem — three kinds of artifacts that combine to let sheets read/write/track data peer-to-peer.

| Artifact | What it does | Lives in |
|---|---|---|
| **Connector** | Module that reads from / writes back to one source sheet via header-resolved schema. ~100–200 LOC. | `templates/connectors/` |
| **Stamper** | Source-side script that auto-stamps stable IDs on new rows. Pasted into the source sheet's own Apps Script. ~130 LOC. | `templates/stampers/` |
| **Tracker** | Full bound-script that consumes connectors + presents a structured UI (Goals, Dashboard, Connections, etc.). ~1500 LOC. | `templates/trackers/` |

This repo is **not deployed**. Files here are paste-sources: copy the relevant templates into a bound Apps Script project (alongside its own Code.js) as needed.

## Why "paste-and-go" rather than a Library

- Tracker owners often aren't developers and shouldn't need clasp / a dev setup.
- Source-sheet owners (who install ID stampers) doubly so.
- No central Library means no shared registry of "who connects to what" — peer-to-peer connections only, gated by Google's per-sheet OAuth grants.

See [`CONNECTOR_INTERFACE.md`](CONNECTOR_INTERFACE.md) for the contract every connector implements.

## Adding a new connector

Use the `/add-crosstrack-connector` Claude Code skill. To install it locally:

```sh
mkdir -p ~/.claude/skills/add-crosstrack-connector
cp skills/add-crosstrack-connector.md ~/.claude/skills/add-crosstrack-connector/SKILL.md
```

Then in Claude Code:

```
/add-crosstrack-connector
```

The skill walks through:
1. What you want to track (plain English).
2. Read-only or write-capable mode (read-only avoids needing a stable ID column).
3. Source URL — schema discovery via google-sheets MCP if available, fallback to "paste the header row."
4. Outputs (printed inline, no disk writes):
   - Connector file ready to drop into your bound script alongside Connector.js
   - Connections row for your tracker's Config tab
   - Optional ID-stamper code (if write-capable and the source has no stable ID)
   - "Allow access" walkthrough

Print-to-screen output by design — the same skill works regardless of whether you have a local repo.

## How a tracker consumes these files

A tracker's bound Apps Script project ends up containing:
1. `Code.js` — the tracker's own orchestration (dashboards, sheet structure, menus).
2. `Connector.js` — copied verbatim from `templates/Connector.template.js`.
3. One or more connector `.js` files — generated by the skill from a chosen source.

Two ways to get those files into the bound Apps Script:

- **Path A (Apps Script editor):** open Extensions ▸ Apps Script on your sheet → File ▸ New ▸ Script file → paste each file's contents. No dev setup required. This is the most common path.
- **Path B (clasp):** if you maintain the tracker locally with git + clasp, drop the files into your tracker repo's `apps_script/` directory and `clasp push`. Same end result; better for version-tracked changes.

Same applies to **ID stampers** (paste-and-go into the source sheet's Apps Script — see [`templates/IdStamper.template.js`](templates/IdStamper.template.js)).

The tracker's `Config` tab has a Connections section where the user pastes a URL per connector. The first time the bound script tries to open that URL, Google's OAuth flow grants access (gated per-sheet, per-user). Connectors read/write directly to the URL.

## What does NOT live here

- Concrete connector instances filled in for specific source sheets — keep those alongside the sheets they connect to.
- Customized stampers (pre-filled for a specific source) — same; they're per-source, not generic.
- Tracker bound-script code itself — that lives in each tracker's own repo (or no repo at all if it was created via Apps Script directly).

## Versioning

- `Connector.template.js` declares `version()`.
- Each connector declares `version`. Start at `0.1.0`; bump deliberately.
- Breaking schema changes bump major version. Consumers can check `Crosstrack.version()` if they want to gate behavior.
